# samba-admin — терминальная веб-панель для Samba

Веб-панель в стиле терминала для управления файловым сервером на Samba:
пользователи, шары, корзина, антивирус, квоты, бэкапы, диски, Active
Directory, уведомления и самообновление — всё из браузера, без необходимости
знать синтаксис `smb.conf`.

Исходники: **https://github.com/lukomsky85/samba-admin-panel**

## Философия и архитектура

Flask-приложение **никогда не работает от root**. Все привилегированные
операции (`useradd`, `smbpasswd`, правка `smb.conf`, монтирование дисков,
`net ads join` и т.д.) идут через один-единственный shell-скрипт
`samba-admin-helper.sh`, который выполняется от root **только** через
`sudo`, и только этот конкретный системный пользователь (`www-panel`) и
только этот конкретный скрипт имеют такое право — прописано в
`/etc/sudoers.d/samba-admin`.

Пароли (вход в панель, смена samba-пароля, администратор AD, SMTP) передаются
в хелпер через **stdin**, а не через аргументы командной строки — иначе они
светились бы в выводе `ps aux` для любого другого пользователя системы.

**Панель всё равно должна жить только в локальной сети** (или за VPN) —
взлом такой панели даёт атакующему возможность создавать системных
пользователей и присоединять сервер к домену. HTTPS, rate-limit на вход и
файрвол (см. «Безопасность») снижают риск, но не заменяют изоляцию по сети.

Установщик (`install.sh`) идемпотентен: можно запускать повторно без
опасений, ничего не задублируется и не сломает то, что уже настроено.

## Возможности

**Пользователи и доступ**
- Создание/удаление Unix-пользователей, смена samba-пароля, включение и
  отключение доступа к шаре без удаления самого пользователя.
- Выбор доступа к конкретной шаре: локальная Unix-группа, группа Active
  Directory, конкретные пользователи AD напрямую, или гостевой доступ без
  пароля — с удобным списком с чекбоксами и поиском (см. «Управление шарами»).

**Шары**
- Ограничение по IP/подсети, запрет записи файлов по расширению.
- Корзина для удалённых файлов (хранить вечно или автоочистка по расписанию).
- Антивирус (ClamAV) на лету при открытии/закрытии файла, с карантином.
- Мониторинговая квота (предупреждение при превышении, не enforced-лимит).
- Бэкап по расписанию (`tar.gz` + ротация) и его восстановление с превью.
- Полный аудит файловых операций через `vfs_full_audit` (кто что открыл/
  удалил/переименовал — не только действия через саму панель).
- Гостевой доступ без пароля — для публичных/анонимных шар, с явным
  предупреждением в интерфейсе про последствия для безопасности.

**Active Directory**
- Присоединение к существующему домену как member server (`winbind`), без
  замены локальных пользователей — оба способа работают параллельно.
- Просмотр списка пользователей/групп AD, проверка соединения, выход из домена.

**Диски**
- Обзор дисков/разделов, здоровье SMART, монтирование уже размеченных
  дисков (с опциональной персистентностью через `/etc/fstab` по UUID).
- Монтирование ISO-образов (только для чтения, через loop-устройство).
- Загрузка ISO прямо из браузера кусками по ~10 МБ с докачкой при обрыве
  связи — рассчитано на образы по 100+ ГБ, а не только для мелких файлов.
- Место на диске по каждой шаре отдельно от занятости всего раздела.

**Мониторинг и уведомления**
- Активные SMB-подключения, журнал действий администратора.
- Уведомления о падении сервисов (`smbd`, `clamav-daemon`, `nginx`,
  сама панель) в Telegram и/или на почту — обе настраиваются прямо в панели.

**Инфраструктура**
- `gunicorn` вместо dev-сервера Flask, HTTPS через `nginx`, файрвол `ufw`,
  автообновления безопасности системы, ротация логов.
- Проверка обновлений и обновление панели прямо с GitHub-релизов.

## Быстрый старт

```bash
git clone https://github.com/lukomsky85/samba-admin-panel.git
cd samba-admin-panel
chmod +x install.sh
sudo ./install.sh
```

Скрипт спросит пароль для входа в панель (Enter — сгенерирует случайный) и
подсеть для файрвола (Enter — пропустит, тогда открой порты руками). Заодно
поставит и настроит ClamAV, nginx с HTTPS, `gunicorn`, ротацию логов и
спросит про включение `ufw` и автообновления безопасности (см. раздел
«Безопасность» — там подробно про то, почему каждый из этих шагов сделан
именно так). В конце выведет ссылку на панель (`https://...`) и пароль —
сохрани его, он больше нигде не показывается в открытом виде (кроме
`/etc/systemd/system/sambapanel.service`, который доступен только root).

Чтобы снести панель (Samba и шары не тронет): `sudo ./uninstall.sh`.

## Управление шарами

- **Создать шару** — создаёт папку (если её нет), настраивает владельца и
  права, добавляет секцию в `/etc/samba/panel-shares.conf`.
- **Убрать шару** — убирает секцию из конфига. **Файлы на диске не трогает**
  — специально, чтобы случайный клик не стёр данные. Удаление данных — руками
  (`rm -rf`), панель этого никогда не делает сама.
- **Read-only / запись** — переключает `writable` без пересоздания шары.
- **Ограничение по IP/подсети** — список адресов/подсетей (например
  `192.168.1.0/24, 10.0.0.5`) через `hosts allow` / `hosts deny = ALL` на
  уровне самой Samba (протокол SMB), а не проверка в веб-приложении. По
  умолчанию (`ALL`) — без ограничений.
- **Запрет записи файлов по расширению** — `veto files` в Samba: файл с
  таким расширением физически невозможно записать, клиент получает отказ
  сразу при попытке создать файл. Уже существующие такие файлы (если
  скопированы до включения запрета) становятся скрытыми и неудаляемыми
  через Samba (`delete veto files = no`) — чистить их, если нужно, придётся
  руками на сервере.
- **Корзина для удалённых файлов** — удаление через шару переносит файл в
  скрытую `.recycle` внутри той же шары (`vfs objects = recycle`, структура
  папок сохраняется). Срок хранения — 0 (навсегда, очистка только руками) или
  N дней (отдельный systemd-таймер `samba-recycle-cleanup.timer`, раз в
  сутки). Кнопка «открыть корзину» показывает список файлов с размером и
  датой и позволяет вернуть **конкретный** файл обратно, а не только очистить
  всё разом — при конфликте путей восстановленный файл получает суффикс
  `.restored-<unix-время>`, ничего не перезаписывается молча.
- **Антивирус (ClamAV)** — сканирование через `vfs_virusfilter` при открытии
  и закрытии файла (и новые загрузки, и уже лежащие файлы при первом
  обращении). Заражённый файл переносится в `.quarantine` и переименовывается
  (не удаляется молча). Это не панацея — сигнатурный антивирус ловит
  известные вирусы, но не целевые атаки; запрет по расширению и антивирус
  дополняют друг друга, а не заменяют.
- **Квота (мониторинговая)** — задаётся в ГБ на шару. **Это не файловая
  система и не enforced-лимит** — Samba не умеет ограничивать объём записи
  без реальных ФС-квот (`vfs objects = quota` не существует — частое
  заблуждение). Панель считает реальный объём (`du`) и подсвечивает
  превышение красным в «Место на диске», но физически ничто не мешает
  записать больше.

  <details>
  <summary>Нужен настоящий enforced-лимит, а не мониторинг? (XFS project quota)</summary>

  ```bash
  # Раздел с шарами должен быть смонтирован с опцией prjquota (XFS)
  # /etc/fstab: /dev/sdX /srv xfs defaults,prjquota 0 0
  sudo mount -o remount /srv

  # Назначить ID проекта папке шары и лимит (пример: id=50, лимит 10ГБ)
  sudo xfs_quota -x -c 'project -s -p /srv/documents 50' /srv
  sudo xfs_quota -x -c 'limit -p bhard=10g 50' /srv
  sudo xfs_quota -x -c 'report -p' /srv
  ```

  Не автоматизировано в панели специально — требует конкретной разметки диска
  (XFS + опция монтирования), которую нельзя безопасно навязать поверх
  произвольной существующей файловой системы.
  </details>

- **Бэкап по расписанию** — ежедневно (`samba-backup.timer`, 02:30 + разброс
  до 20 минут) архивирует содержимое шары (`tar.gz`, без `.recycle` и
  `.quarantine`) в `BACKUP_DEST` из `/etc/sambapanel/backup.conf`, оставляя
  только `RETAIN_COUNT` (по умолчанию 7) самых новых архивов на шару.
  `BACKUP_DEST` может быть чем угодно смонтированным — внешний диск, сетевая
  папка, облачный mount через `rclone`/`s3fs`.
- **Восстановление из бэкапа** (`samba-restore.sh`) — отдельный скрипт,
  **намеренно без кнопки в веб-панели** (та же логика, что и у отказа от
  форматирования дисков — риск перезаписать текущие данные слишком велик для
  одного клика). Без аргументов показывает список архивов; с аргументом —
  сначала превью содержимого (`tar tzf`), затем требует явного подтверждения
  и только потом распаковывает. Это наложение, а не полная синхронизация —
  файлы вне архива не удаляются.
- **Полный аудит файловых операций** — включается по кнопке на конкретной
  шаре через `vfs_full_audit` (не глобально, чтобы не заваливать сервер
  логами со всех шар сразу). Это **очень шумно** (логируется каждая
  VFS-операция, не только удаление/переименование) — намеренно используется
  `full_audit:success/failure = all`, а не конкретные имена операций
  (`unlink`/`unlinkat` и подобные исторически отличались между версиями
  Samba). Пишется через `syslog` на facility `LOCAL5` в отдельный каталог
  `/var/log/samba-full-audit/` (не в `/var/log/samba/` — та директория не
  даёт `rsyslog` прав на создание файла).
- **Выбор типа доступа к шаре** — при создании или редактировании шары можно
  выбрать: обычную локальную Unix-группу (как раньше), группу Active
  Directory, конкретных пользователей AD напрямую (список с поиском и
  чекбоксами, без неочевидного Ctrl/Cmd-клика), или **гостевой доступ без
  пароля**. Технически тип кодируется прямо в значении поля группы
  (`AD:группа`, `ADUSERS:user1,user2`, или буквально `GUEST`) — так не
  пришлось разрастять и без того большую схему `shares.db`. Для AD-группы
  владелец папки резолвится через `getent`/NSS (GID вместо имени), для
  конкретных пользователей папка остаётся `root:root`, доступ регулирует
  только `valid users`. **Ограничение:** имена с пробелами (`DOMAIN\Group
  Name`) пока не поддерживаются — упрощает синтаксис `smb.conf`, где строки
  с пробелами требуют кавычек.

  **Гостевой доступ (`GUEST`)** — `guest ok = yes` в конфиге шары, без
  `valid users`/`force group` вообще (комбинировать их с гостевым доступом
  бессмысленно — гостевая сессия всё равно не проходит проверку по группе).
  Папка получает права `chmod 2777` — доступна на запись **любому локальному
  процессу на сервере**, не только через Samba, потому что гостевое
  Samba-подключение обычно маппится на низкопривилегированную системную
  учётку (`nobody`), а не на реального пользователя, и только мировые права
  доступа гарантируют ей запись. **Единственный способ ограничить, кто может
  подключиться к гостевой шаре — IP/подсеть** (см. выше); пароля или списка
  пользователей для гостевого доступа не существует по определению. Не
  используй для чего-то чувствительного — это буквально публичная шара для
  всех в разрешённой сети.

Все шары живут в отдельном файле `/etc/samba/panel-shares.conf`, который
**полностью перегенерируется** при каждом изменении из файла-состояния
`/etc/sambapanel/shares.db`. Основной `/etc/samba/smb.conf` не трогается —
только подключается через `include`. Перед применением конфиг проверяется
через `testparm`; если невалиден — изменения не применяются, `smbd` не
перезапускается.

**Важно:** после каждого изменения шары панель делает `systemctl restart
smbd`, а не `reload`. `reload` посылает SIGHUP, но уже открытые SMB-сессии
могут жить со старыми правами, пока клиент сам не переподключится — новые
IP-ограничения выглядели бы как «не сработавшие». `restart` сбрасывает
**все** текущие SMB-сессии на сервере, зато новые правила гарантированно
применяются сразу — учитывай это, если шары меняют часто в рабочее время.

## Active Directory (member server)

Присоединяет сервер к уже существующему домену Active Directory как
**member server** через `winbind` — сервер сам **не становится**
контроллером домена (это отдельный, гораздо более тяжёлый сценарий,
`samba-tool domain provision`, который эта панель не покрывает).

**Локальные пользователи и шары не затрагиваются** — AD-интеграция это
дополнительный, независимый способ входа поверх уже работающего локального.

### Что делает присоединение к домену

1. Ставит пакеты `winbind`, `libnss-winbind`, `libpam-winbind`, `krb5-user`
   по требованию (не входит в `install.sh` по умолчанию — не тянуть их
   всем, кому AD не нужен).
2. Пишет `/etc/krb5.conf` (realm, поиск KDC через DNS).
3. Добавляет `winbind` в строки `passwd`/`group` файла `/etc/nsswitch.conf`
   идемпотентно (на Ubuntu 24.04+ может быть уже сделано автоматически).
4. Вставляет AD-блок (`security = ads`, `workgroup`, `realm`, `idmap config`
   с бэкендом `rid`) в секцию `[global]` файла `smb.conf`, между метками
   `# --- BEGIN/END AD INTEGRATION ---`.

   **Важная деталь реализации:** блок вставляется в **конец** секции
   `[global]`, а не в начало. Стандартный `smb.conf` на Ubuntu уже содержит
   свою строку `workgroup = WORKGROUP` — Samba при повторении параметра
   берёт **последнее** значение в секции. Вставка в начало привела бы к
   тому, что штатная строка ниже победила бы наш `workgroup`, и сервер
   продолжил бы считать себя в домене `WORKGROUP` — этот баг был реально
   пойман и исправлен на живом сервере при тестировании.
5. Выполняет `net ads join -U <администратор>` (пароль через stdin).
6. Проверяет успех через **живой `net ads testjoin`**, а не текст вывода
   `net ads join` — на практике `net ads join` иногда рапортует
   обнадёживающее сообщение, даже когда реальный machine trust не
   установился (`Failed to get machine credentials`). Только
   подтверждённый `testjoin` считается успехом.
7. Перезапускает `winbind` и `smbd`.

При любой неудаче (`testparm` не проходит, или `testjoin` после `join` не
подтверждает успех) — `smb.conf` **откатывается** на резервную копию,
сделанную прямо перед изменением.

### Выход из домена

`net ads leave` может завершиться ошибкой (например, если реальное
присоединение изначально не удалось) — **локальная очистка `smb.conf`
происходит в любом случае**, независимо от результата удалённой операции,
чтобы сервер гарантированно вернулся в обычный режим, а не завис в
состоянии «не выйти и не зайти». `krb5.conf` и запись `winbind` в
`nsswitch.conf` не откатываются — осознанно, чтобы не трогать агрессивно
системные файлы, которые могут использоваться чем-то ещё.

### Что проверено, а что нет

Логика вставки/удаления AD-блока в `smb.conf` (идемпотентность, откат при
ошибке, точное попадание именно в `[global]`) **полностью протестирована**
через симуляцию с реальными сценариями сбоя. Генерация `krb5.conf` и
вставка `winbind` в `nsswitch.conf` проверены на реальных системных файлах.

Синтаксис команд сверен с официальной документацией Ubuntu Server (Active
Directory integration). Сам `net ads join` против настоящего контроллера
домена тестировался на реальном сервере в процессе разработки — именно так
и были найдены оба бага выше (порядок параметров, недостаточная проверка
успеха join).

### Известные ограничения (не автоматизировано специально)

- **Регистрация DNS-имени в AD** — если присоединение не проходит из-за
  DNS, официальная документация рекомендует временно выставить hostname в
  FQDN (`sudo hostnamectl hostname host.realm.com`) перед присоединением.
  Панель не делает это автоматически — смена hostname слишком инвазивна для
  автоматического действия.
- **Синхронизация времени** — Kerberos требует расхождения часов с
  контроллером домена не больше 5 минут. Проверь `timedatectl status`.
- **DNS сервера** должен указывать на DNS контроллера домена AD, не на
  публичный DNS — иначе realm не найдётся. Не переключается автоматически.

## Безопасность

### Rate-limit на вход в панель

После каждой неудачной попытки IP получает экспоненциально растущую
задержку: 2с → 4с → 8с → ... до потолка в 5 минут, сбрасывается после 15
минут без новых попыток. Специально через задержку, а не жёсткую
блокировку — блокировка по IP превращается в инструмент DoS против
легитимного админа, если атакующий знает его IP (или это NAT офиса) и
специально долбит панель, чтобы заблокировать настоящего пользователя.

### HTTPS (nginx + самоподписанный сертификат)

Flask слушает **только** `127.0.0.1:5000` — снаружи не виден. Путь к панели
снаружи — только через `nginx` на `443` с TLS (самоподписанный сертификат,
825 дней). Браузер покажет предупреждение о недоверенном сертификате — это
ожидаемо, не баг. Если есть настоящий домен — можно заменить сертификат на
Let's Encrypt.

### Production WSGI-сервер (gunicorn)

`gunicorn` вместо dev-сервера Flask. Конфигурация — **1 процесс + 4 потока**
(`--workers 1 --threads 4 --worker-class gthread`), а не несколько
процессов: rate-limit на вход и сессии хранятся в памяти процесса — с
несколькими воркерами лимит попыток входа фактически умножался бы на их
число. Потоки в одном процессе делят память (лимит и сессии остаются
целостными), конкурентность при этом есть — долгий запрос не блокирует
остальных пользователей панели. `--timeout 120` — с запасом сверх дефолтных
30 секунд для операций подольше (антивирус-скан, `du`, бэкап).

### `ufw` — правила и включение разделены осознанно

Правила `ufw allow` добавляются всегда, но сам `ufw enable` — только по
явному подтверждению в интерактивном режиме, и только после того, как
отдельно разрешён SSH. Включение файрвола молча, без предварительного SSH,
может в одну команду отрезать удалённый доступ к серверу.

### Ротация логов

`audit.log` (журнал действий панели) ротируется просто — Flask открывает
файл заново на каждую запись. `full-audit.log` (`vfs_full_audit`, пишет
`rsyslog`, держит файл открытым постоянно) требует `postrotate` со скриптом
`rsyslog-rotate`, иначе `rsyslog` продолжил бы писать в старый переименованный
файл. Ротация ежедневная (не еженедельная) — при включённом полном аудите
это самый быстрорастущий лог из всех.

### Автообновления безопасности (`unattended-upgrades`)

Опционально при установке (по умолчанию «да» — это касается всей ОС, а не
только Samba). Только пакеты безопасности, не полный `apt upgrade`.

### Уведомления о падении сервисов

Через `OnFailure=` в systemd — при падении `smbd`, `clamav-daemon`, `nginx`,
`sambapanel`, `samba-backup` или `samba-recycle-cleanup` запускается
`samba-notify@<юнит>.service`. Для собственных юнитов правило дописывается
напрямую; для системных пакетных (`smbd.service` и т.д.) — через systemd
drop-in override, чтобы обновление пакета не затёрло правку.

**Настраивается прямо в панели** (вкладка «Мониторинг») — Telegram-токен,
chat_id и/или email, без правки `/etc/sambapanel/notify.conf` через SSH. По
умолчанию пусто — уведомления никуда не уходят, пока не заполнено хотя бы
одно поле. Кнопка «отправить тестовое уведомление» проверяет прямо сейчас,
не дожидаясь реального падения сервиса.

### Настройка SMTP для почтовых уведомлений

`mailutils` + `msmtp` ставятся установщиком; сами данные (сервер, порт,
логин, пароль) заполняются в панели. Конфиг `/etc/msmtprc` получает права
`600` — жёсткое требование самого `msmtp`, иначе он откажется работать.
**Пароль никогда не возвращается в браузер** при повторном открытии —
только флаг «уже задан». Кнопка «отправить тестовое письмо» шлёт реальное
письмо на любой адрес прямо сейчас.

## Наблюдаемость

- **Активные подключения** — сырой вывод `smbstatus` (кто подключён, к
  какой шаре, с какого IP), по кнопке, не автоматически.
- **Место на диске** — отдельно «сколько занимает сама шара» (`du`) и
  «сколько свободно на разделе» (`df`) — это разные числа, если на разделе
  есть что-то ещё кроме шар. `du` по большой шаре небыстрый, поэтому это
  ручная кнопка, а не что-то, что считается на каждое действие.
- **Диски и здоровье** — `lsblk` + SMART (`smartctl`), монтирование уже
  размеченных дисков (не форматирует — риск потери данных слишком велик для
  веб-формы). Не даст отмонтировать диск, если под точкой монтирования есть
  активная шара. Монтирование с опцией «навсегда» добавляет запись в
  `/etc/fstab` по UUID с флагом `nofail`.
- **ISO-образы** — отдельная панель во вкладке «Диски»: поиск `.iso` файлов
  в указанной папке и монтирование через loop-устройство. Всегда **только
  для чтения** (`mount -o loop,ro`) — это не ограничение панели, а
  физическое свойство формата ISO9660, в готовый образ нельзя дописать
  данные. Смонтированные ISO появляются в общем списке дисков (тип `loop`)
  и отключаются той же кнопкой «отмонтировать», что и обычные диски.
  **Без персистентности** — в отличие от обычных дисков, монтирование ISO
  не добавляется в `/etc/fstab` автоматически (сознательно, чтобы не
  плодить edge-case'ы вокруг loop-монтирования при загрузке); переживает
  только до перезагрузки или ручного отключения. Поиск `.iso`-файлов и
  сама команда монтирования проверены на реальном образе, но именно вызов
  `mount` до конца не проверялся в среде, где писалась эта функция (там нет
  модуля ядра `iso9660` — типичное ограничение контейнеров); на обычном
  Ubuntu-сервере этот модуль есть из коробки.
- **Загрузка ISO из браузера** — та же панель, форма ниже поиска. Файл
  режется в браузере на куски по ~10 МБ (`File.slice()`), каждый кусок —
  отдельный запрос с указанием смещения в байтах. Обрыв связи не требует
  начинать заново: страница (даже после обновления) сама находит
  незавершённую загрузку того же файла (по имени+размеру+папке назначения)
  и продолжает с последнего успешно принятого байта. Повторная отправка
  одного и того же куска безопасна — сервер пишет по смещению (`seek`), а
  не всегда в конец файла, так что случайный дубль не портит и не удваивает
  данные.

  Куда именно пишутся байты — определяется автоматически: если панель
  (процесс `www-panel`) уже может писать напрямую в папку назначения
  (например, шара с гостевым доступом, `chmod 2777`) — пишет сразу туда,
  финализация — мгновенное переименование на месте. Если нет — использует
  общую служебную папку `/opt/sambapanel/uploads` (её `install.sh` готовит
  и отдаёт `www-panel` заранее), а финальный перенос на место с правильными
  правами делает привилегированный хелпер одним вызовом в самом конце — не
  на каждый кусок, иначе на файле в 100+ ГБ это было бы невыносимо медленно.
  Свободное место на диске проверяется **до** начала загрузки (и в общей
  папке, и в конечной, если они разные — при непрямой записи временно нужно
  места в обоих местах сразу).

  Протестировано сквозным сценарием через реальные HTTP-запросы (не
  выдумано): докачка после «обрыва связи» с точного места остановки,
  повторная отправка одного и того же куска без порчи файла, побайтовое
  совпадение итогового файла с оригиналом, и корректный отказ при заведомой
  нехватке места на диске. Путь **без** прямой записи (через
  привилегированный `iso_finalize_upload`) проверен на уровне самого
  bash-хелпера отдельно — сквозной прогон именно через `sudo` в этой среде
  не поднимался (нет реального пользователя `www-panel` с настроенными
  правами), так что первый реальный аплоад в такую шару стоит один раз
  проверить глазами.
- **Журнал действий администратора** — кто/что/когда через панель
  (создание пользователя, смена прав и т.д.). При одном общем пароле на
  панель «кто» — это IP, не имя человека.
- **Полный аудит файловых операций** — см. «Управление шарами» выше.

## Обновление панели

Во вкладке «Мониторинг» есть блок «Обновление панели»:

1. **«Проверить обновления»** — только чтение, сравнивает тег последнего
   релиза на GitHub (`GET /repos/lukomsky85/samba-admin-panel/releases/latest`)
   с локальным файлом `/opt/sambapanel/VERSION`. Ничего не скачивает.
2. **«Обновить до vX.Y.Z»** (появляется только если найдена версия новее) —
   скачивает архив конкретного релиз-тега и запускает **его собственный
   `install.sh`** поверх текущей установки — тот же самый идемпотентный
   установщик, что и при первой установке, а не отдельный менее проверенный
   путь. Текущий пароль панели читается из уже установленного systemd-юнита
   и передаётся установщику, чтобы после обновления логин остался тем же.

   Запускается **асинхронно**, в фоновом процессе, отвязанном от текущего
   HTTP-запроса — сама переустановка идёт несколько минут и в процессе
   `install.sh` делает `systemctl restart sambapanel`, что оборвало бы
   синхронный запрос. Прогресс — кнопка «показать журнал обновления»
   (`/var/log/sambapanel/update.log`).

Если «проверить обновления» показывает `error` с текстом про `rate limit` —
это не баг панели, просто GitHub API ограничивает частоту
неаутентифицированных запросов с одного IP (обычно 60/час); подожди и
попробуй снова.

## Установка руками (если нужно понять или поменять шаги)

`install.sh` — источник истины: он идемпотентен, безопасен для повторного
запуска и уже проверен. Ручные шаги ниже — для понимания, из чего состоит
установка, а не полноценная альтернатива (например, они не покрывают
ClamAV, nginx/HTTPS, `gunicorn`, ротацию логов, `ufw`, уведомления и
`unattended-upgrades` — для всего этого проще прочитать соответствующие,
короткие и читаемые секции самого `install.sh`).

```bash
# Подключить конфиг шар, управляемых панелью, к основному smb.conf
sudo touch /etc/samba/panel-shares.conf
# ВАЖНО: include должен попасть В СЕКЦИЮ [global], а не в конец файла
# вслепую — если ниже есть другие секции (например [printers]), простой
# `tee -a` в конец файла воткнёт include не туда. install.sh делает это
# правильно; вручную — открой smb.conf и добавь строку внутри [global].

# Системный пользователь для сервиса (без прав администратора)
sudo useradd -r -m -d /opt/sambapanel -s /usr/sbin/nologin www-panel

# Скопировать проект
sudo mkdir -p /opt/sambapanel
sudo cp -r app.py templates VERSION /opt/sambapanel/
sudo chown -R www-panel:www-panel /opt/sambapanel

# Хелпер-скрипт и sudoers-правило
sudo cp samba-admin-helper.sh /usr/local/sbin/samba-admin-helper.sh
sudo chmod 750 /usr/local/sbin/samba-admin-helper.sh
sudo cp samba-admin.sudoers /etc/sudoers.d/samba-admin
sudo chmod 440 /etc/sudoers.d/samba-admin
sudo visudo -c

# Python-окружение
sudo -u www-panel python3 -m venv /opt/sambapanel/venv
sudo -u www-panel /opt/sambapanel/venv/bin/pip install flask gunicorn

# systemd unit — задать пароль панели и секрет сессий вручную
sudo cp sambapanel.service /etc/systemd/system/sambapanel.service
sudo nano /etc/systemd/system/sambapanel.service
# заменить CHANGE_ME на пароль панели,
# CHANGE_ME_SECRET на 'python3 -c "import secrets; print(secrets.token_hex(32))"'
sudo systemctl daemon-reload
sudo systemctl enable --now sambapanel

# Открыть порт только для LAN
sudo ufw allow from 192.168.1.0/24 to any port 443 proto tcp
```

## Чеклист перед продакшеном

```bash
sudo systemctl status sambapanel
sudo journalctl -u sambapanel -f

sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh list_users
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh list_shares
# ^ оба должны сработать без пароля и без ошибок — sudoers настроен верно

sudo systemctl list-timers samba-recycle-cleanup.timer
sudo systemctl start samba-recycle-cleanup.service
sudo journalctl -u samba-recycle-cleanup -n 20

sudo systemctl status clamav-daemon
ls -la /var/run/clamav/clamd.ctl

sudo nginx -t
curl -k https://127.0.0.1/login       # -k из-за самоподписанного сертификата

ps aux | grep gunicorn                # процессы gunicorn, не "python3 app.py"
sudo journalctl -u sambapanel -n 20 | grep -i "using worker"  # "gthread"

sudo ufw status verbose               # Status: active, видны правила 443/445

sudo logrotate -d /etc/logrotate.d/sambapanel
sudo logrotate -d /etc/logrotate.d/samba-full-audit

sudo /usr/local/sbin/samba-restore.sh test
# ^ без второго аргумента — список архивов шары "test" (если бэкап настроен)

systemctl status unattended-upgrades
cat /etc/apt/apt.conf.d/20auto-upgrades

sudo systemctl start samba-notify@test-unit.service
sudo journalctl -t samba-notify -n 5

which mail && which msmtp
cat /etc/msmtprc
ls -la /etc/msmtprc                   # права -rw------- (600)

# --- Active Directory (только если присоединялись) ---
timedatectl status
resolvectl status
sudo net ads testjoin
sudo wbinfo -t
id "EXAMPLE\\jammy"

# --- Обновления ---
cat /opt/sambapanel/VERSION
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh check_update
```

## Известные ограничения и планы на будущее

- **Отдельные именные логины** для нескольких администраторов — сейчас
  один общий пароль на панель, «кто» в журнале действий — это IP, не имя.
- **Уведомления о событиях внутри панели** (не о падении сервисов — это уже
  сделано) — например, при заполнении корзины, срабатывании антивируса,
  превышении квоты или отказе SMART-диска. Инфраструктура уже есть
  (`samba-notify-failure.sh` + `notify.conf`), не хватает только вызовов
  из `samba-admin-helper.sh` в нужных местах.
- **Настоящие enforced-квоты** через XFS project quota — см. врезку в
  «Управление шарами», если мониторинговой квоты недостаточно.
- **Форматирование дисков** через панель — сознательно не сделано (см.
  «Наблюдаемость» → «Диски и здоровье») из-за риска потери данных.
- **AD-группы/пользователи с пробелами** в имени (`DOMAIN\Group Name`) —
  пока не поддерживаются при выборе доступа к шаре, чтобы не связываться с
  кавычками в `smb.conf` для строк с пробелами. Без пробелов — работает.
