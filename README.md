# samba-admin — терминальная веб-панель для Samba-шары

Маленькая панель в стиле терминала: создаёт Linux-пользователей, включает/выключает
им доступ к Samba-шаре и меняет samba-пароли — всё через браузер в локальной сети.

## Как это устроено (важно понимать перед установкой)

Flask-приложение **никогда не работает от root**. Все привилегированные операции
(`useradd`, `smbpasswd`, `usermod`) идут через один-единственный shell-скрипт
`samba-admin-helper.sh`, который выполняется от root **только** через `sudo`,
и только этот конкретный пользователь (`www-panel`) и только этот конкретный
скрипт имеют такое право — прописано в `/etc/sudoers.d/samba-admin`.

Пароли передаются в хелпер через stdin, а не через аргументы командной строки,
чтобы они не светились в выводе `ps aux` для других пользователей системы.

**Панель всё равно должна жить только в локальной сети** (или за VPN) —
взлом такой панели даёт атакующему возможность создавать системных
пользователей. Rate-limit и HTTPS (см. раздел «Безопасность» ниже) снижают
риск, но не заменяют изоляцию по сети.

## Active Directory (member server)

Вкладка «Active Directory» позволяет присоединить сервер к уже существующему
домену Active Directory как **member server** через `winbind` — сервер сам
**не становится** контроллером домена, это отдельный, гораздо более тяжёлый
сценарий (`samba-tool domain provision`), который эта панель не покрывает.

**Локальные пользователи и шары не затрагиваются.** AD-интеграция — это
дополнительный, независимый способ входа поверх уже работающего локального.
Можно продолжать создавать локальных Unix-пользователей через панель как
раньше, а AD-пользователи получают доступ параллельно, без конфликта.

### Что делает присоединение к домену

1. Ставит пакеты `winbind`, `libnss-winbind`, `libpam-winbind`, `krb5-user`
   (если ещё не установлены — не входит в `install.sh` по умолчанию, чтобы
   не тянуть их всем, кому AD не нужен).
2. Пишет `/etc/krb5.conf` (realm, поиск KDC через DNS).
3. Добавляет `winbind` в строки `passwd`/`group` файла `/etc/nsswitch.conf`
   (на Ubuntu 24.04+ это может быть уже сделано автоматически — проверяется
   идемпотентно, не дублирует).
4. Вставляет AD-блок конфигурации (`security = ads`, `workgroup`, `realm`,
   `idmap config` с бэкендом `rid` — стандартный выбор, когда в схеме AD не
   расширены атрибуты POSIX/RFC2307) в секцию `[global]` файла `smb.conf`,
   между метками-комментариями `# --- BEGIN/END AD INTEGRATION ---`.
5. Выполняет `net ads join -U <администратор>` (пароль передаётся через
   stdin, не через аргументы командной строки — не светится в `ps aux`,
   как и все пароли в этом проекте).
6. Перезапускает `winbind` и `smbd`.

При любой ошибке на шаге 4–5 (`testparm` не проходит, `net ads join` не
подтвердил успех) — `smb.conf` **откатывается** на резервную копию,
сделанную прямо перед изменением.

### Что проверено, а что нет (важно!)

Логика вставки/удаления AD-блока в `smb.conf` (включая идемпотентность,
откат при ошибке, и то, что блок вставляется именно в `[global]`, а не
куда попало — в этом проекте уже был баг именно такого рода с `include`,
попавшим в `[print$]`) **полностью протестирована** через симуляцию: точная
вставка после `[global]`, идемпотентность при повторном присоединении,
и побайтовое совпадение конфига после отката через удаление блока по меткам.
Генерация `krb5.conf` и вставка `winbind` в `nsswitch.conf` тоже проверены
на реальных системных файлах.

**Сам `net ads join` против настоящего контроллера домена проверить было
физически нечем** — в среде, где это писалось, нет доступа к живому AD.
Синтаксис команд сверен с официальной документацией Ubuntu Server (раздел
Active Directory integration, обновлён в июне 2026), но реальное
присоединение стоит для начала попробовать на тестовом сервере/виртуалке,
а не сразу на проде, особенно в первый раз.

### Известные ограничения (не автоматизировано специально)

- **Регистрация DNS-имени сервера в AD** — во время `net ads join` Samba
  пытается зарегистрировать хост в DNS AD. Если IP сервера ещё не
  зарегистрирован, официальная документация рекомендует временно выставить
  hostname в полное доменное имя (`sudo hostnamectl hostname
  host.realm.com`) перед присоединением и вернуть обратно после. Панель
  **не делает это автоматически** — смена hostname системы слишком
  инвазивная операция, чтобы делать её без явного, отдельного согласия;
  если присоединение не удаётся из-за DNS — попробуй эту команду руками.
- **Синхронизация времени** — Kerberos требует расхождения часов с
  контроллером домена не больше 5 минут. Панель это не проверяет и не
  чинит — убедись, что `chrony`/`systemd-timesyncd` работает и показывает
  разумное время (`timedatectl status`).
- **DNS сервера** должен указывать на DNS контроллера домена AD (не на
  публичный DNS) — иначе `realm`/домен просто не найдётся. Тоже не
  проверяется и не переключается автоматически (сетевые настройки —
  слишком системная вещь, чтобы трогать её из веб-панели без явного шага).
- **Использование AD-групп прямо в качестве группы доступа к шаре** (чтобы
  при создании шары выбрать `EXAMPLE\Accounting` вместо локальной
  `sharegroup`) — в этой версии **не реализовано**. Список пользователей и
  групп AD можно посмотреть во вкладке (кнопки «обновить» под «Пользователи
  AD» / «Группы AD»), но использовать их напрямую в `valid users` шары
  пока нужно руками, отредактировав `panel-shares.conf` (с учётом, что
  панель перегенерирует этот файл целиком при следующем изменении шары —
  см. «Управление шарами» выше).
- **Выход из домена** не откатывает `krb5.conf` и запись `winbind` в
  `nsswitch.conf` — осознанно, чтобы не трогать агрессивно системные файлы,
  которые могут использоваться чем-то ещё помимо Samba.

## Управление шарами

Панель теперь умеет не только пользователей, но и сами шары:

- **Создать шару** — создаёт папку (если её нет), ставит `chown группа:группа` +
  `chmod 2775`, и добавляет секцию в `/etc/samba/panel-shares.conf`.
- **Убрать шару** — убирает секцию из конфига. **Файлы на диске не трогает** —
  специально, чтобы случайный клик не стёр данные. Если нужно удалить и данные,
  делай это руками (`rm -rf`), панель этого не делает никогда.
- **Read-only / запись** — переключает `writable` для шары без пересоздания.
- **Ограничение по IP/подсети** — для каждой шары можно указать список
  IP-адресов или подсетей (например `192.168.1.0/24, 10.0.0.5`), которым
  разрешён доступ. Реализовано через `hosts allow` / `hosts deny = ALL`
  на уровне самой Samba, а не в веб-приложении — это работает на уровне
  протокола SMB, надёжнее любой проверки в Flask. По умолчанию (`ALL`) —
  без ограничений, доступ определяется только паролем/группой как раньше.
- **Запрет записи файлов по расширению** — можно запретить создание файлов
  с определёнными расширениями в конкретной шаре (например `exe, bat, cmd,
  scr, vbs, msi, ps1` — типичные носители вирусов). Реализовано через `veto
  files` в Samba: файл с таким расширением физически невозможно записать —
  клиент получит отказ прямо при попытке создать файл, а не просто увидит
  предупреждение. Существующие такие файлы (если были скопированы до
  включения запрета) становятся скрытыми и их не получится удалить через
  Samba (`delete veto files = no`) — удалять такие файлы, если нужно,
  придётся руками на сервере.
- **Корзина для удалённых файлов** — при включении удаление файла через шару
  не стирает его, а переносит в скрытую папку `.recycle` внутри той же шары
  (`vfs objects = recycle` в Samba, структура папок сохраняется). Для каждой
  шары настраивается срок хранения:
  - **0 дней** — хранить вечно, очистка только руками (кнопка «очистить
    корзину» в панели, показывает количество файлов и требует подтверждения).
  - **N дней** — отдельный systemd-таймер (`samba-recycle-cleanup.timer`,
    ставится установщиком, запускается раз в сутки) удаляет из `.recycle`
    файлы старше N дней автоматически. Он не трогает шары с сроком «навсегда».
- **Антивирус (ClamAV)** — при включении файлы сканируются самим Samba через
  `vfs_virusfilter` (модуль встроен в Samba с версии 4.8, ставится пакетом
  `samba-vfs-modules`) при открытии и при закрытии файла — то есть и новые
  загрузки, и уже лежащие в шаре файлы будут проверены при первом же обращении
  к ним. Заражённый файл переносится в скрытую `.quarantine` внутри шары и
  переименовывается (не удаляется молча — если понадобится разобраться, что
  именно нашли, файл никуда не пропал). Очистить карантин — отдельная кнопка
  в панели, требует подтверждения, безвозвратно. Если антивирус и корзина
  включены одновременно, сначала работает проверка антивирусом, и только
  потом — логика корзины при удалении.
  **Важно:** это не панацея — сигнатурный антивирус ловит известные вирусы,
  но не защищает от целенаправленных атак или свежих угроз. Запрет по
  расширению (veto files) и антивирус дополняют друг друга, а не заменяют.
- **Квота (мониторинговая)** — задаётся в ГБ на шару. **Это не файловая
  система и не enforced-лимит** — Samba не умеет ограничивать объём записи
  без реальных ФС-квот. Панель считает реальный объём шары (`du`) в панели
  «Место на диске» и подсвечивает превышение красным, но физически ничто не
  мешает записать больше — квота информирует, а не блокирует. Если нужен
  настоящий жёсткий лимит — см. врезку про XFS project quota ниже.
- **Бэкап по расписанию** — при включении шара попадает в ежедневный обход
  `samba-backup.timer` (02:30 ночи + случайный разброс до 20 минут, чтобы не
  бить бэкапы разных серверов в одну секунду, если их несколько в одной сети).
  Каждый прогон делает `tar.gz` содержимого шары (без `.recycle` и
  `.quarantine` — это служебный мусор панели, не данные пользователя) в
  папку `BACKUP_DEST` из `/etc/sambapanel/backup.conf`, и оставляет только
  `RETAIN_COUNT` (по умолчанию 7) самых новых архивов на шару — старые
  удаляются автоматически. `BACKUP_DEST` может быть чем угодно смонтированным:
  внешний диск, сетевая папка, облачный mount через `rclone`/`s3fs` — скрипту
  всё равно, физически куда он пишет файлы. Если у шары нет свежего бэкапа
  (таймер ещё не запускался, или `BACKUP_DEST` не настроен) — панель честно
  покажет «включён, ещё не запускался», а не наврёт про несуществующую дату.

<details>
<summary>Если нужен настоящий enforced-лимит места, а не мониторинг (XFS project quota)</summary>

Samba не имеет собственного механизма квот (`vfs objects = quota` не
существует — распространённое заблуждение). Единственный способ реально
заблокировать запись при превышении лимита — квоты на уровне файловой
системы. Проще всего это делается через **XFS project quota**, которая
умеет ограничивать конкретную ДИРЕКТОРИЮ (не обязательно весь раздел):

```bash
# Раздел, где лежат шары, должен быть смонтирован с опцией prjquota (XFS)
# Пример для /etc/fstab: /dev/sdX /srv xfs defaults,prjquota 0 0
# После правки fstab: sudo mount -o remount /srv

# Назначить ID проекта папке шары и лимит (пример: id=50, лимит 10ГБ)
sudo xfs_quota -x -c 'project -s -p /srv/documents 50' /srv
sudo xfs_quota -x -c 'limit -p bhard=10g 50' /srv

# Проверить
sudo xfs_quota -x -c 'report -p' /srv
```

Это не автоматизировано в панели специально — требует конкретной разметки
диска (XFS, опция монтирования), которую нельзя безопасно навязать поверх
произвольной существующей файловой системы без риска что-то сломать.

</details>

Все шары, созданные через панель, живут в отдельном файле `/etc/samba/panel-shares.conf`,
который **полностью перегенерируется** при каждом изменении из файла-состояния
`/etc/sambapanel/shares.db`. Основной `/etc/samba/smb.conf` эта логика не трогает —
только подключает его через `include`. Перед применением конфиг проверяется
через `testparm`; если он невалиден, изменения не применяются и smbd не перезагружается.

**Важно:** после каждого изменения шары (создание, смена прав, IP-ограничения)
панель делает `systemctl restart smbd`, а не `reload`. Это специально — `reload`
посылает конфигу SIGHUP, но уже открытые SMB-подключения могут продолжать жить
со старыми правами до тех пор, пока клиент сам не переподключится, из-за чего
новые ограничения по IP выглядят как «не сработавшие». `restart` сбрасывает
**все** текущие SMB-сессии на сервере (не только к изменяемой шаре), так что
новые правила гарантированно применяются сразу — но это значит, что у всех,
кто в этот момент активно копирует файлы через Samba, соединение на секунду
оборвётся. Для домашнего/офисного сервера это некритично, но если шары меняют
часто в рабочее время — учитывай этот момент.

## Безопасность

### Rate-limit на вход в панель

После каждой неудачной попытки входа IP получает экспоненциально растущую
задержку до следующей попытки: 2с → 4с → 8с → 16с... до потолка в 5 минут.
Счётчик привязан к IP и сбрасывается, если 15 минут не было новых попыток.

Специально сделано через задержку, а не через жёсткую блокировку на N минут —
жёсткая блокировка по IP превращается в удобный инструмент DoS против
легитимного пользователя, если атакующий просто знает его IP (или это NAT,
за которым сидит весь офис) и специально долбит панель его паролем, чтобы
заблокировать доступ настоящему админу. Задержка не даёт этого сделать —
максимум, чего добьётся атакующий, — сам будет ждать по 5 минут между попытками.

Счётчик живёт в памяти процесса панели — если её перезапустить (`systemctl
restart sambapanel`), счётчик сбросится. Для одного процесса Flask (как
здесь) это ожидаемо и не проблема.

### Антивирус (ClamAV)

Включается индивидуально для каждой шары (см. раздел «Управление шарами»).
Устанавливается и настраивается автоматически через `install.sh`:
`clamav-daemon`, `clamav-freshclam`, `samba-vfs-modules`. Сокет ClamAV
(`/var/run/clamav/clamd.ctl`) специально сделан доступным для записи/чтения
всем локальным процессам (`LocalSocketMode 666`) — иначе Samba не смогла бы
подключаться к нему от имени только что созданных пользователей (заранее
неизвестно, кого панель создаст, чтобы добавить в группу `clamav`). Сам
ClamAV не отдаёт наружу ничего, кроме вердикта «заражён/не заражён», так что
это приемлемый компромисс на локальном сервере — но если сервер когда-нибудь
станет мультиарендным (недоверенные пользователи с shell-доступом), это стоит
пересмотреть.

Первое обновление баз (`freshclam`) может занять несколько минут — установщик
это ждёт до 30 секунд, но если не успеет, доустановится в фоне сам
(`clamav-freshclam.service` работает как таймер).

### HTTPS (nginx + самоподписанный сертификат)

Flask-приложение слушает **только** `127.0.0.1:5000` — снаружи оно вообще не
видно. Единственный путь к панели снаружи — через `nginx` на порту `443` с
TLS. `install.sh` сам генерирует самоподписанный сертификат (`openssl req -x509`,
срок 825 дней) и настраивает `nginx` как обратный проксирующий сервер.

Браузер при первом заходе покажет предупреждение «сертификат не доверенный» —
это ожидаемо для самоподписанного сертификата и не баг. Прими риск («Advanced»
→ «Proceed anyway» в Chrome, аналогично в других браузерах) — трафик всё равно
шифруется, предупреждение просто означает, что сертификат не подписан
публичным доверенным центром (для локального сервера без домена это нормально
и практически неизбежно без покупки/выпуска настоящего сертификата).

Если у сервера есть настоящий домен и вы хотите избавиться от предупреждения —
можно заменить `/etc/ssl/sambapanel/cert.pem` и `key.pem` на сертификат от
Let's Encrypt (`certbot`) и перезапустить `nginx`; сама панель этого не касается.

Порт `5000` снаружи больше не имеет смысла открывать (Flask туда изнутри и не
слушает) — `install.sh` открывает в `ufw` только `443` и `445`.

### Production WSGI-сервер (gunicorn)

Панель запускается через `gunicorn`, а не через встроенный dev-сервер Flask
(`app.run()`), который сам прямо предупреждает: *"Do not use it in a
production deployment."* Конфигурация — **1 процесс (`--workers 1`) + 4
потока (`--threads 4`, worker-class `gthread`)**, а не несколько процессов.
Это осознанный выбор, а не недосмотр: rate-limit на вход в панель и
сессии хранятся в обычном Python-словаре в памяти процесса — с несколькими
процессами (воркерами) у каждого была бы своя копия этого счётчика, и лимит
попыток входа фактически умножался бы на число воркеров (пароль можно было
бы подбирать в N раз быстрее). Потоки в пределах одного процесса делят
память между собой, так что и rate-limit, и куки сессий остаются
целостными, а конкурентность всё равно есть — долгий запрос (например,
подсчёт места на диске) не блокирует остальных пользователей панели.

`--timeout 120` — с запасом сверх дефолтных 30 секунд `gunicorn`, чтобы
операции подольше (антивирус-скан, `du` по большой шаре, создание бэкапа)
не обрывались по таймауту прямо посреди работы.

### `ufw` — правила и включение разделены осознанно

Установщик добавляет правила `ufw allow` всегда, но включает сам `ufw`
(`ufw enable`) только по явному подтверждению в интерактивном режиме — и
только после того, как отдельно разрешит SSH (`ufw allow OpenSSH`). Причина:
если у `ufw` есть правило, но сам он выключен — оно просто ничего не делает,
неопасно. А если включить `ufw` МОЛЧА, не позаботившись заранее про SSH,
можно в одну команду отрезать себе же удалённый доступ к серверу и потерять
над ним контроль. Установщик после всех правил явно проверяет статус `ufw` и
громко предупреждает, если он всё ещё выключен — правила есть, а толку от
них ноль, пока `ufw` неактивен.

### Ротация логов

`/etc/logrotate.d/sambapanel` — журнал действий панели (`audit.log`).
Flask открывает файл заново на каждую запись (не держит дескриптор
постоянно открытым), поэтому обычной ротации (переименование + новый
пустой файл) достаточно — следующая запись просто откроет по тому же пути
уже новый файл.

`/etc/logrotate.d/samba-full-audit` — лог `vfs_full_audit`. Этот файл пишет
`rsyslog`, который держит его открытым постоянно, поэтому тут обязателен
`postrotate` со скриптом `rsyslog-rotate` (идёт в комплекте с пакетом
`rsyslog`) — без него после ротации `rsyslog` продолжил бы писать в старый,
уже переименованный и невидимый по прежнему пути файл. Ротация — ежедневная
(не еженедельная, как у `audit.log`) и с более коротким сроком хранения
(14 дней) — при включённом «полном аудите» это самый быстрорастущий лог
из всех в этом проекте.

### Восстановление шары из бэкапа (`samba-restore.sh`)

`samba-backup.sh` создаёт архивы, но раньше не было ни одного
задокументированного или автоматизированного способа их восстановить —
бэкап, который никто не пробовал восстановить, по факту не бэкап.
`samba-restore.sh` — отдельный скрипт (устанавливается в
`/usr/local/sbin/`), который:
- без аргументов показывает список доступных архивов конкретной шары с
  размером и датой;
- с аргументом `<имя_шары> <файл_архива>` или `<имя_шары> latest` сначала
  показывает **превью** содержимого архива (`tar tzf`, ничего не меняя на
  диске), затем требует явного подтверждения (ввести `да`), и только
  после этого распаковывает архив поверх текущего содержимого шары.

Восстановление — это **наложение**, а не полная синхронизация: файлы из
архива перезаписывают одноимённые файлы в шаре, но файлы, которых нет в
архиве (появившиеся после бэкапа), не удаляются.

**Намеренно нет кнопки «восстановить» в веб-панели** — та же логика, что и
у отказа от форматирования дисков через панель: это операция с реальным
риском перезаписать текущие данные, ей место в команде с явным
подтверждением в терминале, а не в одном клике мышкой, где случайный клик
не туда обходится дорого. Проверено на реальном сценарии: файл удалён и
файл испорчен → восстановлен `samba-restore.sh` → оба вернулись к
состоянию на момент бэкапа.

### Автообновления безопасности (`unattended-upgrades`)

Устанавливается опционально при установке (спрашивает явно, по умолчанию
«да», раз это касается всей системы, а не только Samba/панели). Настраивает
`APT::Periodic::Unattended-Upgrade` — автоматически подтягиваются только
пакеты безопасности (origin из стандартного шаблона пакета
`unattended-upgrades`, не полный `apt upgrade` всего подряд). Проверить,
что реально включено: `cat /etc/apt/apt.conf.d/20auto-upgrades` и
`systemctl status unattended-upgrades`.

### Уведомления о падении сервисов (Telegram/почта)

Через `OnFailure=` в systemd — если `smbd`, `clamav-daemon`, `nginx`,
`sambapanel`, `samba-backup` или `samba-recycle-cleanup` завершаются с
ошибкой, автоматически запускается `samba-notify@<имя-юнита>.service`,
который вызывает `/usr/local/sbin/samba-notify-failure.sh` с именем упавшего
юнита. Для своих собственных юнитов (`sambapanel.service` и т.д.) правило
дописывается прямо в файл; для системных пакетных юнитов (`smbd.service`,
`clamav-daemon.service`, `nginx.service`) — через systemd drop-in override
в `/etc/systemd/system/<юнит>.d/samba-notify-override.conf`, а не прямой
правкой файла юнита, потому что обновление соответствующего пакета
перезаписало бы прямую правку при следующем `apt upgrade`.

**Оба канала (Telegram и почта) настраиваются прямо в панели** — вкладка
«Мониторинг», без необходимости редактировать `/etc/sambapanel/notify.conf`
руками через SSH. Панель читает и пишет тот же файл, ручное редактирование
остаётся рабочим способом, просто больше не единственным.

**По умолчанию уведомления никуда не уходят** — `/etc/sambapanel/notify.conf`
устанавливается пустым шаблоном; скрипт, не найдя ни Telegram-токена, ни
адреса почты, просто тихо пишет в `journal` и ничего не отправляет (это
безопасное поведение по умолчанию, а не недоделанная фича — установщик
не может сам придумать тебе Telegram-бота или настроить почтовый релей).

Проверено: скрипт корректно не считает отправку успешной, если Telegram
вернул ошибку (например, неверный токен) — используется `curl -f`, без
которого `curl` засчитывает НТТР-ошибку как «успех», раз соединение вообще
состоялось (это отдельно проверено на реальном ответе прокси/API).
Протестировать вручную, не дожидаясь реального падения сервиса:
`sudo systemctl start samba-notify@test-unit.service` — прогонит весь
конвейер уведомления с фиктивным именем юнита. В панели то же самое —
кнопка «отправить тестовое уведомление».

### Настройка SMTP для почтовых уведомлений (через панель)

`install.sh` ставит `mailutils` (даёт команду `mail`) и `msmtp` (лёгкий
SMTP-клиент, который реально доставляет письма через внешний почтовый
сервер — без него `mail` был бы установлен, но письма никуда бы не уходили).
Сами данные (SMTP-сервер, порт, адрес отправителя, логин, пароль) —
заполняются в панели, во вкладке «Мониторинг» → «Настройка отправки почты
(SMTP)», а не через ручную правку `/etc/msmtprc`.

Технические детали:
- Панель пишет `/etc/msmtprc` и **обязательно** ставит права `600` —
  это не наша прихоть, а жёсткое требование самого `msmtp`: если конфиг
  читаем кем-то, кроме владельца, `msmtp` отказывается им пользоваться
  вообще, с явной ошибкой.
- **Пароль никогда не возвращается обратно в браузер** при повторном
  открытии вкладки — только флаг «уже задан / не задан». Чтобы поменять
  остальные поля (например, адрес отправителя), не обязательно вводить
  пароль заново — если оставить поле пароля пустым при сохранении, старый
  сохранённый пароль остаётся как есть. Проверено: обновление одного поля
  с пустым паролем не стирает ранее сохранённый.
- Кнопка «отправить тестовое письмо» шлёт реальное письмо на указанный
  адрес прямо сейчас — не нужно ждать настоящего падения сервиса, чтобы
  проверить, что почта в принципе доходит.
- Если сервер, с которого уходит почта, никогда раньше не отправлял писем
  с этого адреса — есть шанс, что первое письмо уйдёт в папку «Спам» у
  получателя, это нормально для новых отправителей, а не признак поломки.

## Установка на Ubuntu — одной командой

```bash
chmod +x install.sh
sudo ./install.sh
```

Скрипт спросит пароль для входа в панель (Enter — сгенерирует случайный) и
подсеть для файрвола (Enter — пропустит, тогда открой порты руками). Заодно
поставит и настроит ClamAV, nginx с HTTPS, `gunicorn`, ротацию логов и
спросит про включение `ufw` (см. раздел «Безопасность» выше — там подробно
про то, почему каждый из этих шагов сделан именно так). В конце выведет
ссылку на панель (`https://...`) и пароль — сохрани его, он больше нигде не
показывается в открытом виде (кроме `/etc/systemd/system/sambapanel.service`,
который доступен только root).

Скрипт идемпотентен — можно перезапускать без опасений, ничего не задублирует.

Чтобы снести панель (Samba и шары не тронет): `sudo ./uninstall.sh`.

## Установка руками (если нужно понять/поменять шаги)

**Примечание:** шаги ниже покрывают панель и Samba, но НЕ включают установку
и настройку ClamAV и nginx/HTTPS — для них проще прочитать сам `install.sh`
(секции про ClamAV и HTTPS там короткие и читаемые) и повторить нужные
команды руками, либо просто запустить `install.sh` целиком — он идемпотентен
и не поломает то, что уже настроено вручную по шагам ниже.

```bash
# 0. Убедись, что Samba уже настроена (см. предыдущие шаги — smbd, sharegroup, /srv/share)

# 0.1 Подключить конфиг шар, управляемых панелью, к основному smb.conf
sudo touch /etc/samba/panel-shares.conf
echo 'include = /etc/samba/panel-shares.conf' | sudo tee -a /etc/samba/smb.conf
sudo testparm -s   # должно пройти без ошибок

# 1. Системный пользователь для самого сервиса (без прав администратора)
sudo useradd -r -m -d /opt/sambapanel -s /usr/sbin/nologin www-panel

# 2. Скопировать проект
sudo mkdir -p /opt/sambapanel
sudo cp -r app.py templates static /opt/sambapanel/
sudo chown -R www-panel:www-panel /opt/sambapanel

# 3. Установить хелпер-скрипт
sudo cp samba-admin-helper.sh /usr/local/sbin/samba-admin-helper.sh
sudo chmod 750 /usr/local/sbin/samba-admin-helper.sh
sudo chown root:root /usr/local/sbin/samba-admin-helper.sh

# 4. Установить sudoers-правило
sudo cp samba-admin.sudoers /etc/sudoers.d/samba-admin
sudo chmod 440 /etc/sudoers.d/samba-admin
sudo visudo -c    # проверить, что синтаксис не сломан

# 5. Python-окружение
sudo apt install -y python3-venv
sudo -u www-panel python3 -m venv /opt/sambapanel/venv
sudo -u www-panel /opt/sambapanel/venv/bin/pip install flask

# 6. Задать пароль входа в панель и установить systemd unit
sudo cp sambapanel.service /etc/systemd/system/sambapanel.service
sudo nano /etc/systemd/system/sambapanel.service   # заменить CHANGE_ME на реальный пароль

sudo systemctl daemon-reload
sudo systemctl enable --now sambapanel

# 7. Открыть порт ТОЛЬКО для LAN (пример подсети 192.168.1.0/24 — поправь под свою)
sudo ufw allow from 192.168.1.0/24 to any port 5000 proto tcp
```

Дальше открываешь `http://IP-сервера:5000` из браузера в локальной сети,
вводишь пароль панели — и работаешь.

## Что умеет панель

- **Создать пользователя** — делает `useradd`, ставит unix-пароль, добавляет
  в группу `sharegroup`, ставит samba-пароль. Пользователь сразу может
  подключаться к шаре.
- **Сменить samba-пароль** — если человек забыл пароль от шары (не от Linux).
- **Включить/выключить доступ к шаре** — не удаляя пользователя из системы,
  просто убирает его из `sharegroup`.
- **Удалить пользователя** — полностью, включая домашнюю папку.
- Живой лог — каждая команда и её результат видны в консоли на странице,
  как в настоящем терминале.
- **Сводная панель сверху** — количество шар, пользователей (активных/всего),
  файлов в корзинах и в карантине суммарно по всем шарам, одним взглядом,
  без прокрутки. Обновляется бесплатно из уже загруженных данных (не делает
  отдельных запросов) — считается на клиенте из того, что и так пришло при
  обновлении списков шар/пользователей.
- **Восстановление файла из корзины** — кнопка «открыть корзину» показывает
  список файлов внутри `.recycle` (с размером и датой удаления) и позволяет
  вернуть конкретный файл обратно в шару, а не только очистить корзину
  целиком. Если по тому же пути в шаре уже что-то лежит — восстановленный
  файл получит суффикс `.restored-<unix-время>`, ничего не перезаписывается
  молча. Имена файлов передаются между сервером и панелью в base64 — так
  восстановление работает даже для файлов с пробелами, кириллицей или
  спецсимволами в имени, не ломая протокол внутренних API-вызовов.
- **Поиск по шарам и пользователям** — поле фильтра над каждым списком,
  фильтрует уже загруженные строки на клиенте по подстроке в имени, без
  обращения к серверу. Полезно, когда шар/пользователей становится много и
  плоский список неудобно листать.

## Наблюдаемость

### Активные подключения

Панель «Активные подключения» дёргает `smbstatus` от root (через тот же
хелпер) и показывает сырой вывод: кто подключён, с какого IP, к какой шаре,
с какого момента. Не парсится и не упрощается специально — формат `smbstatus`
информативнее любой попытки его причесать, и так проще сверять с тем, что
видно в `journalctl`, если придётся разбираться в проблеме глубже. Обновляется
по кнопке, не сама по себе — чтобы не дёргать `smbstatus` без необходимости.

### Место на диске

Кнопка «проверить место на диске» считает для каждой шары:
- сколько байт занимает сама шара (`du -sb` по её пути),
- сколько всего/свободно/занято (в процентах) на разделе, где эта шара лежит.

Это осознанно два разных числа — если на разделе есть что-то ещё кроме шар
(система, другие данные), «занято шарой» и «занято на разделе» не совпадут.
Полоска подсвечивается жёлтым при 75%+ занятости раздела и красным при 90%+.

`du` по большой шаре может быть небыстрым (реально сканирует файлы на диске),
поэтому это отдельная кнопка, а не что-то, что считается на каждый чих —
иначе создание шары или переключение прав стало бы неожиданно медленным.

### Диски и здоровье (SMART), монтирование

Панель «Диски и здоровье» показывает физические диски и разделы (`lsblk`),
их здоровье (`smartctl` из пакета `smartmontools`) и позволяет монтировать
уже размеченные диски.

**Намеренно НЕ делает панель:**
- **Не форматирует диски.** Создание разделов и файловой системы — ручная
  операция через SSH, один раз при добавлении нового диска в сервер.
  Ошибка в веб-форме, стирающая данные без возврата, — слишком серьёзный
  риск для кнопки в браузере. Если диск размечен и на нём есть файловая
  система (`lsblk` показывает `FSTYPE`), панель может её примонтировать;
  если нет — покажет понятную ошибку вместо попытки угадать, что делать.
- **Не отмонтирует диск, если под ним есть активная шара.** Проверяется по
  путям в базе шар: если путь шары совпадает с точкой монтирования или
  лежит внутри неё — отмонтирование отклоняется с именами конфликтующих
  шар, а не превращает шару в путь в никуда.

**SMART-здоровье:** быстрая сводка (`smartctl -H -A`) даёт статус
PASSED/FAILED и температуру по каждому физическому диску; кнопка «SMART
подробно» — полный вывод `smartctl -a` для конкретного диска (количество
переназначенных секторов, часы наработки и т.д.). `FAILED` — сигнал
готовиться к замене диска ДО того, как он откажет и заберёт данные с собой;
многие бытовые/облачные диски вообще не поддерживают SMART (виртуальные
диски в облаке, USB-мосты без прозрачной проброски SMART-команд) — тогда
будет честное «нет данных SMART», а не ложное «всё хорошо».

**Монтирование:** выбираешь неразмеченный (точнее — размеченный, но не
смонтированный) раздел, задаёшь точку монтирования, опционально — галку
«подключать автоматически при загрузке». Если включена, панель добавляет
строку в `/etc/fstab` **по UUID** (а не по `/dev/sdX`, который может
поменяться местами между дисками после перезагрузки) с флагом `nofail` —
если диск вдруг физически отсутствует при следующей загрузке, сервер всё
равно загрузится, а не зависнет в ожидании отсутствующего диска.

### Журнал действий

Каждое изменяющее действие через панель (создание/удаление пользователя,
шары, смена прав, паролей, входы в панель — успешные и неуспешные) пишется
в `/var/log/sambapanel/audit.log`: время, IP, результат (OK/FAIL), сама
команда. Read-only запросы (списки, статус подключений, место на диске) в
журнал не попадают — иначе он бы захлёбывался обновлениями страницы вместо
того, чтобы показывать реальные изменения.

**Важно понимать ограничение:** сейчас у панели один общий пароль на всех
администраторов — значит, «кто» в журнале — это IP-адрес, а не конкретное
имя человека. Если панелью пользуется больше одного человека и нужно точно
знать, кто из них что сделал, — это единственное реальное ограничение
текущей схемы, и решается только добавлением отдельных именных логинов
(что не входит в эту версию, но architecturally к этому легко прийти:
достаточно хранить несколько пар логин/пароль вместо одной переменной
`SAMBAPANEL_PASSWORD` и класть `session["user"]` при входе).

Журнал пишет процесс панели (`www-panel`), а не root — файл лежит в
`/var/log/sambapanel/`, который установщик создаёт с владельцем `www-panel`.
Ротацию логов установщик не настраивает — если беспокоит рост файла на
активно используемом сервере, добавь `/etc/logrotate.d/sambapanel` руками.

### Полный аудит файловых операций (кто реально трогал файл в шаре)

Журнал действий выше показывает только то, что делали ЧЕРЕЗ ПАНЕЛЬ (создал
пользователя, включил антивирус). Он не покажет, что конкретный пользователь
скачал, переименовал или удалил файл прямо через Проводник/Finder — для
этого включается отдельно, на уровне конкретной шары, через Samba-модуль
`vfs_full_audit`.

**Это ОЧЕНЬ шумно.** `full_audit:success/failure = all` логирует буквально
каждую VFS-операцию — не только удаление/переименование, но и открытие,
закрытие, чтение метаданных и т.д. Мы намеренно используем `all`, а не
конкретные имена операций (`unlink`, `rename` и т.п.) — конкретные имена
исторически отличались между версиями Samba (например, в некоторых версиях
используются `unlinkat`/`renameat` вместо `unlink`/`rename`), и `all` —
единственный вариант, который официально задокументирован и гарантированно
работает одинаково на любой версии. Компромисс — шум вместо риска, что
конкретное имя операции окажется невалидным на конкретной версии Samba
(ровно так уже ловили баг с `virusfilter:quarantine directory` раньше в этом
же проекте — здесь решили не рисковать тем же способом второй раз).

Технически: Samba пишет эти сообщения через `syslog(3)` на facility `LOCAL5`
(это не файл сам по себе — без отдельного правила в `rsyslog` сообщения ушли
бы в общий `/var/log/syslog` вперемешку со всем остальным). `install.sh`
сам добавляет `/etc/rsyslog.d/49-samba-full-audit.conf`, который направляет
`LOCAL5` в `/var/log/samba-full-audit/full-audit.log` и останавливает
дальнейшую обработку (`& stop`), чтобы сообщения не дублировались в общий
syslog. Эта папка — **специально не** `/var/log/samba` (там лежат обычные
логи Samba): та директория `root:adm` с правами `rwxr-x---`, то есть группе
`adm` доступно только чтение, а `rsyslog` (работает от пользователя
`syslog`, который состоит в группе `adm`) должен именно **создать** файл —
для этого нужны права на запись в саму папку, которых там нет. Поэтому
`/var/log/samba-full-audit/` создаётся отдельно и сразу с владельцем
`syslog:adm`, чтобы `rsyslog` мог писать в неё без проблем.

Включается по кнопке на конкретной шаре (не глобально — специально, чтобы
не заваливать сервер логами со всех шар сразу, если это не нужно). Панель
показывает последние 200 строк в разделе «Аудит файловых операций» — читает
файл через тот же root-хелпер, что и всё остальное, так что права доступа
к самому файлу лога не имеют значения (в отличие от `/var/log/sambapanel/`,
здесь нет нужды делать `www-panel` совладельцем чего-либо).

Если нужно сузить список операций (меньше шума, но с риском версийной
несовместимости конкретных имён) — правь `full_audit:success`/`failure`
руками в сгенерированном `panel-shares.conf` для конкретной шары, зная, что
при следующем изменении шары через панель твоя правка будет затёрта
(конфиг перегенерируется целиком, см. «Управление шарами» выше).

## Обновление панели с GitHub

Исходники: **https://github.com/lukomsky85/samba-admin-panel** (ссылка есть
в шапке панели и в блоке обновлений). Файл `VERSION` в корне репозитория —
источник истины для текущей установленной версии; `install.sh` копирует его
в `/opt/sambapanel/VERSION` при установке/обновлении.

**Проверка обновлений** (кнопка «проверить обновления») — только чтение,
дёргает `GET https://api.github.com/repos/lukomsky85/samba-admin-panel/releases/latest`
и сравнивает тег последнего релиза с содержимым локального `VERSION`.
Ничего не скачивает и не меняет на этом шаге.

**Применение обновления** (кнопка «обновить до vX.Y.Z», появляется только
если найдена более новая версия) — скачивает конкретный релиз-тег с GitHub
(`archive/refs/tags/<tag>.tar.gz`) и запускает **его собственный `install.sh`**
поверх текущей установки — тот же самый идемпотентный установщик, что и при
первой установке, а не отдельный, менее проверенный путь для апдейта.
Текущий пароль панели читается из уже установленного systemd-юнита и
передаётся установщику, чтобы `install.sh` не сгенерировал новый пароль —
после обновления логин остаётся тем же.

Запускается **асинхронно** (в фоновом процессе, отвязанном от текущего HTTP-
запроса) — сама переустановка занимает несколько минут (пакеты, systemd) и
в процессе `install.sh` делает `systemctl restart sambapanel`, что оборвало
бы сам запрос, если бы обновление шло синхронно внутри текущего процесса
панели. Прогресс смотри кнопкой «показать журнал обновления»
(`/var/log/sambapanel/update.log`).

**Чтобы выпустить новый релиз** для этой системы обновлений: подними версию
в файле `VERSION` (например `1.1.0`), закоммить, и создай GitHub Release с
тегом, СОВПАДАЮЩИМ с содержимым `VERSION` (`git tag v1.1.0` — тег должен
быть именно тем текстом, который сравнивается со строкой в `VERSION`, так
что либо храни в `VERSION` значение с `v` в начале, либо тегируй без `v` —
главное, чтобы оба совпадали дословно). Проверено вживую против реального
GitHub API (упёрлись в rate limit при тестировании — обработка этого
случая, в отличие от гипотетической, подтверждена настоящим ответом API,
не выдуманным).

## Проверка перед продакшеном

```bash
sudo systemctl status sambapanel     # сервис запущен?
sudo journalctl -u sambapanel -f     # смотреть логи в реальном времени
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh list_users
# ^ должно сработать без пароля и без ошибок — проверка, что sudoers настроен верно

sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh list_shares
# ^ то же самое для шар

sudo systemctl list-timers samba-recycle-cleanup.timer
# ^ должен показать время следующего запуска автоочистки корзины
sudo systemctl start samba-recycle-cleanup.service  # запустить очистку прямо сейчас, не дожидаясь таймера
sudo journalctl -u samba-recycle-cleanup -n 20      # что реально удалила последняя очистка

sudo systemctl status clamav-daemon   # антивирус поднялся?
ls -la /var/run/clamav/clamd.ctl      # сокет существует?
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh set_share_antivirus test yes
# ^ должно сработать без ошибки "ClamAV не запущен", если демон и сокет на месте

sudo nginx -t                         # конфиг nginx валиден?
curl -k https://127.0.0.1/login       # -k нужен из-за самоподписанного сертификата
# ^ должна вернуться HTML-страница логина, а не ошибка соединения

sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh active_connections
# ^ должен показать вывод smbstatus без запроса пароля
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh disk_usage
# ^ должен показать DISK|... строки по каждой шаре

ls -la /var/log/sambapanel/           # владелец должен быть www-panel
cat /var/log/sambapanel/audit.log     # появляется после первого действия в панели

cat /etc/sambapanel/backup.conf       # BACKUP_DEST задан? (если бэкап настроен)
sudo systemctl start samba-backup.service   # запустить бэкап прямо сейчас
sudo journalctl -u samba-backup -n 30       # что реально забэкапилось
ls -la "$(grep BACKUP_DEST /etc/sambapanel/backup.conf | cut -d'"' -f2)"
# ^ должны появиться архивы <шара>-<дата>.tar.gz для шар с включённым бэкапом

cat /etc/rsyslog.d/49-samba-full-audit.conf   # правило маршрутизации на месте?
sudo systemctl status rsyslog                 # active (running)?
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh set_share_full_audit test yes
# ^ включить на тестовой шаре, затем зайти в неё с любого клиента и открыть файл
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh file_audit_log
# ^ должны появиться строки вида "имя_шары|user|IP|..." после операций с файлами

sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh list_block_devices
# ^ должен показать BLOCKDEV|... строки по всем дискам/разделам
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh disk_smart_summary
# ^ должен показать SMART|... по каждому физическому диску (PASSED/FAILED/UNSUPPORTED)

ps aux | grep gunicorn                # должны быть видны процессы gunicorn, не "python3 app.py"
sudo journalctl -u sambapanel -n 20 | grep -i "using worker"
# ^ должно быть "Using worker: gthread" — проверка, что запущено правильным воркер-классом

sudo ufw status verbose               # Status: active, и видны нужные правила (443, 445)
# ^ если Status: inactive — правила есть, но ничего не блокируют/не разрешают,
#   включи явно: sudo ufw enable (сначала убедись, что SSH разрешён!)

sudo logrotate -d /etc/logrotate.d/sambapanel        # без ошибок парсинга?
sudo logrotate -d /etc/logrotate.d/samba-full-audit  # без ошибок парсинга?
sudo logrotate -f /etc/logrotate.d/sambapanel        # принудительно прогнать ротацию прямо сейчас
ls -la /var/log/sambapanel/                          # должен появиться audit.log.1.gz (если лог не пуст)

sudo /usr/local/sbin/samba-restore.sh test
# ^ без второго аргумента — должен показать список архивов шары (или сообщение
#   "архивов не найдено", если бэкап для неё ещё не запускался ни разу)

systemctl status unattended-upgrades   # active (running) или (exited) — оба ок
cat /etc/apt/apt.conf.d/20auto-upgrades

sudo systemctl start samba-notify@test-unit.service
sudo journalctl -t samba-notify -n 5
# ^ если notify.conf не заполнен — увидишь "уведомление НЕ отправлено", это
#   ожидаемо; заполни notify.conf и прогони снова, чтобы проверить реальную отправку

which mail && which msmtp             # оба должны быть установлены
cat /etc/msmtprc                      # появится после первого сохранения SMTP-настроек в панели
ls -la /etc/msmtprc                   # права должны быть -rw------- (600), иначе msmtp откажется работать
echo "проверка" | mail -s "тест" твой@адрес.ru
tail -20 /var/log/msmtp.log           # если письмо не дошло — причина обычно тут (неверный пароль, порт, TLS)

# --- Active Directory (только если решил присоединяться) ---
timedatectl status                    # время синхронизировано? (важно для Kerberos)
resolvectl status                     # DNS указывает на сервер AD, а не на публичный DNS?
sudo net ads testjoin                 # после присоединения — "Join is OK"?
sudo wbinfo -t                        # проверка доверенного секрета
id "EXAMPLE\\jammy"                   # резолвится ли реальный AD-пользователь через NSS?

cat /opt/sambapanel/VERSION           # версия установлена?
sudo -u www-panel sudo -n /usr/local/sbin/samba-admin-helper.sh check_update
# ^ должен вернуть UPDATECHECK|... — либо uptodate, либо available с тегом,
#   либо error с понятным текстом (rate limit / нет релизов / нет интернета)
```

## Что стоит добавить самому, если будешь развивать дальше

- Отдельные именные логины для нескольких администраторов (сейчас один общий
  пароль на панель — см. ограничение в разделе «Журнал действий» выше).
- Уведомления о **событиях внутри самой панели** (не о падении сервисов —
  это уже сделано, см. раздел «Уведомления о падении сервисов» выше) —
  например, при заполнении корзины, срабатывании антивируса, превышении
  квоты или отказе SMART-диска. Инфраструктура для этого уже есть
  (`samba-notify-failure.sh` + `notify.conf`), не хватает только вызовов
  из `samba-admin-helper.sh` в нужных местах.
- Настоящие enforced-квоты через XFS project quota — см. врезку в разделе
  «Управление шарами» выше, если мониторинговой квоты недостаточно.
- Форматирование новых дисков через панель — сознательно не сделано (см.
  раздел «Диски и здоровье») из-за риска необратимой потери данных при
  ошибке в веб-форме; делается руками через SSH.
- AD-группы/пользователи с **пробелами** в имени (`DOMAIN\Group Name`) —
  сейчас сознательно не поддерживаются при выборе доступа к шаре (см.
  «Управление шарами» → выбор AD-группы/пользователей), чтобы не связываться
  с кавычками в `smb.conf` для строк с пробелами. Обычные AD-группы/логины
  без пробелов работают полностью.
